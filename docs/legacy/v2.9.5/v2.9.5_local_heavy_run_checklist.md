# v2.9.5 Local Heavy-Run Checklist

Date: 2026-04-06
Status: Active operator checklist for local `full` runs

This checklist defines the operator-owned boundary for local heavy runs.
For `v2.9.5`, these runs are intentionally outside the required CI set:

- benchmark full on real libraries
- production full on real libraries

`required / ci-sized-full-fixture` only proves that `full` mode remains executable on
a small deterministic fixture. It does not prove real-data full-run readiness.

## Operator checklist

Run these checks before treating a local heavy run as usable evidence:

1. confirm the intended role-safe command line matches the target config role
2. confirm `comparison_type` matches the role policy and is not being read as an algorithm-comparison claim
3. confirm managed theta, PAT diagnostics, caps, and assays point to the intended run-specific fixtures
4. confirm the output directory is clean or that any pre-existing manifest has been reviewed for stale-artifact guard failures
5. run the local full command and retain the console banner, warnings, and summary
6. confirm `run_manifest.json`, `output_inventory.json`, `rule3_trace_summary.json`, `theta_rule1_resolution.json`, and `replay_audit.json` all materialize
7. confirm `replay_audit.result == "PASS"` or document the blocker if it stays `UNCLEAR`
8. confirm Cap truth-source reconciliation stays verified when Cap is requested
9. record elapsed time and any fallback / skip / warn messages in the local audit memo

## Non-goal of required CI

The required CI set must not be read as a substitute for these local checks.
CI green means the frozen contract, guards, and deterministic fixtures remain healthy.
It does not mean:

- benchmark full real-data is equivalent to production full real-data
- production full has been exercised on the current library outside the operator machine
- local runtime envelope or interruption-resume behavior has been revalidated
