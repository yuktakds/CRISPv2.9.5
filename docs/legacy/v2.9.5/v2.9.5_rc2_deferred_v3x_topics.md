# v2.9.5 rc2 Deferred v3.x Topics

Date: 2026-04-06
Status: active deferral index for the rc2 release line

This document separates semantic design topics from the `v2.9.5` rc2 scope.
Its purpose is simple:

- keep rc2 focused on release hardening
- stop semantic debate from flowing back into the frozen rc2 line
- make the handoff to `v3.x` explicit and reviewable

## rc2 boundary

Allowed on the `v2.9.5` rc2 line:

- bug fixes that preserve frozen semantics
- docs and release-engineering work
- tests and CI maintenance
- additive shell diagnostics and guards

Not allowed on the rc2 line without resetting scope:

- object-logic reinterpretation
- taxonomy or comparison semantics changes
- proposal-connected Rule3 behavior
- benchmark / production meaning changes

## Deferred topic index

| Topic | Why it is deferred | Why it is not rc2 scope |
| --- | --- | --- |
| Rule3 proposal-connected evolution | Requires a new ADR, replay-evidence contract, and semantic policy version | rc2 keeps Rule3 observational only |
| Any same-pose style requirement | Needs a new execution-significant definition and validation envelope | rc2 cannot add a new object-logic predicate quietly |
| CoreSCV reverse-flow / backflow issues | Requires an explicit decision on whether shell diagnostics may influence Core outcomes | rc2 preserves the current responsibility split |
| Object-logic changes | Changes the frozen release meaning, not just shell hardening | rc2 is release engineering, not semantic redesign |
| Taxonomy / comparison semantics changes | Would redefine `same-config` or `cross-regime` interpretation | rc2 keeps benchmark / production separation fixed |
| Benchmark / production redefinition | Would alter operator expectations, CI interpretation, and audit claims | rc2 relies on the current role-safe boundary |

## Entry rule for v3.x discussion

Do not reopen a deferred topic on the rc2 line unless all of the following are true:

1. the desired semantic delta is written before coding
2. the affected artifact / replay contract is enumerated up front
3. benchmark and cross-regime guard implications are stated explicitly
4. the work is moved to a `v3.x` branch or ADR track

## Operational consequence

If a future change request matches any topic in the table above, the default answer is:

- do not extend rc2 scope
- open or update the relevant `v3.x` ADR / design note instead
