# CRISP v2.9.5 Manifest and Inventory Schema Freeze

Date: 2026-04-03
Status: Active freeze policy for `H-4`

This document fixes the artifact-schema policy for `v2.9.5`.
Its purpose is to stop post-RC drift in the integrated run contract.

## Scope

The freeze applies to these payloads:

1. `run_manifest.json`
2. `output_inventory.json`
3. `completion_checks_json`
4. `completion_basis_json`

The freeze covers key names, value meanings, and version anchors.

## Version anchors

| Artifact / payload | Version anchor in `v2.9.5` | Source of truth |
| --- | --- | --- |
| `run_manifest.json` | top-level `spec_version == "v2.9.5"` | `crisp/v29/manifest.py` |
| `output_inventory.json` | inherits the integrated artifact contract from `run_manifest.spec_version == "v2.9.5"` | `crisp/v29/manifest.py` |
| `completion_checks_json` | explicit `schema_version == "v1"` | `crisp/v29/manifest.py` |
| `completion_basis_json` | inherits the integrated artifact contract from `run_manifest.spec_version == "v2.9.5"` | `crisp/v29/cli.py` |

`output_inventory.json` and `completion_basis_json` do not have standalone explicit schema
version fields in `v2.9.5`. That is a deliberate description of the current implementation,
not an omission in this document.

## Frozen policy by artifact

### `run_manifest.json`

Frozen for `v2.9.5`:

- `spec_version` stays `v2.9.5`
- existing top-level keys remain present and keep the same meaning
- Rule1 managed-theta metadata fields remain:
  - `theta_rule1_table_id`
  - `theta_rule1_table_version`
  - `theta_rule1_table_digest`
  - `theta_rule1_table_source`
  - `theta_rule1_runtime_contract`
- `completion_basis_json` remains the nested basis record for comparison, PathYes, skip, and output-materialization semantics

Not allowed after RC without version bump or RC reset:

- removing a top-level key
- renaming a top-level key
- changing the type of an existing key
- reinterpreting an existing key with incompatible semantics

### `output_inventory.json`

Frozen for `v2.9.5`:

- current top-level keys remain present and keep the same meaning:
  - `run_id`
  - `run_mode`
  - `requested_branches`
  - `implemented_branches`
  - `generated_outputs`
  - `missing_outputs`
  - `schema_validation`
  - `warnings`
  - `run_mode_complete`
  - `branch_status_json`
  - `completion_basis_json`
  - `completion_checks_json`
  - `repo_root_source`
  - `repo_root_resolved_path`
- `schema_validation.status` continues to use `PASS | WARN | FAIL`
- `completion_checks_json` remains the machine-readable completion truth source embedded in inventory

Not allowed after RC without version bump or RC reset:

- removing or renaming any top-level inventory key
- changing how `run_mode_complete` is derived
- diverging inventory completion semantics from replay-audit completion semantics

### `completion_checks_json`

Frozen for `v2.9.5`:

- `schema_version == "v1"`
- the required key set remains:
  - `schema_version`
  - `required_outputs`
  - `materialized_output_paths`
  - `required_branches`
  - `declared_missing_outputs`
  - `missing_output_files`
  - `empty_output_files`
  - `completion_blocker_outputs`
  - `required_branch_statuses`
  - `incomplete_required_branches`
  - `schema_hard_errors`
  - `schema_warnings`
  - `schema_validation_status`
  - `run_mode_complete`

Not allowed after RC without version bump or RC reset:

- changing `schema_version` away from `v1`
- removing a required key
- changing the meaning of completion blockers, missing outputs, or required branch statuses

### `completion_basis_json`

Frozen for `v2.9.5`:

- comparison metadata remains:
  - `comparison_type`
  - `comparison_type_source`
- PathYes / Rule1 gating metadata remains:
  - `pathyes_mode_requested`
  - `pathyes_force_false_requested`
  - `pathyes_mode_resolved`
  - `pathyes_state_source`
  - `pathyes_diagnostics_status`
  - `pathyes_diagnostics_error_code`
  - `pathyes_diagnostics_source`
  - `pathyes_goal_precheck_passed`
  - `pathyes_rule1_applicability`
  - `pathyes_skip_code`
- execution and output provenance remains:
  - `skip_reason_codes`
  - `output_fallback_reason_codes`
  - `output_materialization_events`
  - `required_outputs_by_mode`
  - `required_branches_by_mode`
- `phase0_core_only` and `run_mode` keep their current meaning

Not allowed after RC without version bump or RC reset:

- changing the meaning of `comparison_type` or `comparison_type_source`
- changing PathYes fields so that validation and replay audit no longer interpret them identically
- changing `skip_reason_codes` semantics without updating all report contracts and replay audit logic together under a new version

## Allowed change class after RC

The only safe default is additive, documented, and low-risk change.

Allowed without resetting the RC judgement:

- docs-only clarifications
- tests and CI-only updates
- bug fixes that preserve the frozen key set and semantics

Allowed only with an explicit version bump or RC reset:

- adding a new key that downstream tooling must understand to preserve correctness
- any removal, rename, type change, or semantic reinterpretation

## Sidecar interpretation

Additive sidecars such as `theta_rule1_resolution.json` and `rule3_trace_summary.json`
do not extend the frozen top-level inventory schema on the `v2.9.5` line.

Safe treatment:

- record the sidecar path through `generated_outputs`
- keep `required_outputs_by_mode` focused on completion blockers
- document replay / validator behavior separately

Unsafe treatment:

- adding new top-level inventory keys just to catalog sidecars
- silently promoting sidecars into completion blockers without a versioned contract change

The standing sidecar catalog is documented in
[v2.9.5_sidecar_artifact_policy.md](v2.9.5_sidecar_artifact_policy.md).

## Interpretation rule

For `v2.9.5`, the schema freeze is primarily semantic, not cosmetic.
Field order, JSON formatting, and path-local values may vary.
Key presence, value meaning, and replay/validation interpretation may not drift.

## Release implication

This freeze is the contract referred to by [v2.9.5_rc_checklist.md](v2.9.5_rc_checklist.md).
If a future change cannot fit inside this policy, it should not ship as a quiet `v2.9.5`
patch to the RC line.
