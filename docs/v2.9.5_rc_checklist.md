# CRISP v2.9.5 RC Checklist and Release Judgement

Date: 2026-04-03
Status: Active checklist for `H-2`

This document fixes the entry criteria for calling `v2.9.5-rc1`.
It is intentionally narrow: `v2.9.5` is in release engineering, not feature development.
As of 2026-04-06, the current line is designated `v2.9.5-rc1`.

## RC entry rule

Call the current line `v2.9.5-rc1` only when all of the following are true:

1. `uv run pytest -q` passes on the candidate commit.
2. the required integrated CI set defined in [../audit/v2.9.5_ci_required_matrix.md](../audit/v2.9.5_ci_required_matrix.md) is green and flakiness remains acceptable under `V-1`.
3. benchmark, production, and CI-sized full integrated fixtures remain replay-auditable.
4. the schema freeze in [v2.9.5_manifest_inventory_schema_freeze.md](v2.9.5_manifest_inventory_schema_freeze.md) is accepted as the standing artifact contract.
5. post-RC changes can be limited to bug fixes, docs, and tests unless the candidate is reset.

If any item is false, the line stays pre-RC.

## Post-RC change policy

After `v2.9.5-rc1`, the default allowed changes are:

- bug fixes that do not alter the frozen artifact contract
- docs updates
- tests and CI-only changes
- RC revisions that preserve the frozen object logic and artifact semantics

The following are not allowed without resetting the RC judgement or bumping the versioned contract:

- renaming, removing, or retyping keys in `run_manifest.json`
- renaming, removing, or retyping keys in `output_inventory.json`
- changing `completion_checks_json.schema_version` from `v1` without a documented schema migration
- changing the meaning of `completion_basis_json` keys used by validation or replay audit
- changing config taxonomy semantics
- changing object-logic semantics on the `v2.9.5` line
- moving PAT-backed data into Core object-logic predicates
- turning Rule3 trace-only behavior into proposal-connected behavior without a new ADR

If a future change needs semantic reinterpretation rather than defect correction,
it must be isolated to a `v3.x` line instead of being merged into `v2.9.5-rc1`.

## Required CI Set

The standing required CI set is exactly:

- `required / benchmark-integrated-smoke`
- `required / production-integrated-smoke`
- `required / ci-sized-full-fixture`
- `required / config-guard-matrix`
- `required / replay-inventory-crosscheck`
- `required / cap-artifact-invariants`
- `required / v2.9.5-matrix`

`benchmark full` and `production full` on real libraries are not part of this required set.
They remain operator-owned local heavy runs and must be evaluated with
[v2.9.5_local_heavy_run_checklist.md](v2.9.5_local_heavy_run_checklist.md).

## Release judgement states

Use exactly one of the following decisions for an RC audit:

- `blocked`: a release-stopping contract, stability, or policy issue exists
- `continue`: candidate is not blocked, but dry-run evidence or release paperwork is still incomplete
- `release`: candidate is acceptable for tagging as `v2.9.5-rc1`
